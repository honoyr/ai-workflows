# Upgrading OpenClaw v2026.4.15 → v2026.4.24: the breaking-change map

Upgrading across this nine-release window is **not** a routine bump — it crosses a Plugin SDK removal, a major shed of default dependencies, several security-tightened defaults that will lock out previously-working configurations, and at least four mandatory `openclaw doctor --fix` migrations. The single most disruptive change is in v2026.4.24, which removes `api.registerEmbeddedExtensionFactory(...)` from the Plugin SDK and stops shipping `node-llama-cpp`, `pdfjs-dist`, the Anthropic Vertex SDK, and Bonjour discovery as core dependencies. Layered on top, v2026.4.20 hardened workspace `.env` handling and gateway broadcast scopes, v2026.4.21 made owner-command authorization strict, and v2026.4.22 removed the Codex CLI auth import path entirely. Plan for a re-auth pass, a config audit, and at least one `openclaw doctor --fix` cycle before declaring the upgrade clean.

A note on coverage: OpenClaw uses calendar versioning (`vYYYY.M.D`), so "4.15" and "4.24" refer to **v2026.4.15** (Apr 14, 2026) and **v2026.4.24** (Apr 25, 2026). Between 4.15 stable and 4.20 stable the project shipped primarily through beta tags (e.g. `v2026.4.19-beta.2`); detailed standalone notes for 4.16–4.19 are not separately published, and operators upgrading from 4.15 effectively inherit all 4.20 changes as their first stable bump.

## What will actually break your install

The hardest stops are concentrated in two releases. **In v2026.4.24, `api.registerEmbeddedExtensionFactory(...)` is removed from the Plugin SDK** — every third-party plugin that rewrites tool results (a common pattern for Pi-embedded transforms) must migrate to `api.registerAgentToolResultMiddleware(...)` and declare `contracts.agentToolResultMiddleware` with the targeted harnesses. The new middleware runs uniformly across Pi and Codex app-server dynamic tools, but the old call site simply no longer exists. In the same release, **`node-llama-cpp` is no longer installed by default**; local-embedding setups silently lose capability unless operators install it explicitly as the optional runtime package. The 4.24 release also extracts `pdfjs-dist` and PDF image-rendering deps into a new bundled `document-extract` plugin, moves the **Anthropic Vertex SDK** into the Anthropic provider plugin, and pushes `@homebridge/ciao` (LAN Bonjour discovery) into its own bundled plugin. Bare-core or aggressively pruned installs will lose PDF understanding, Vertex routing, and gateway LAN discovery on first restart.

The other hard break is **v2026.4.22's removal of the Codex CLI auth import path**. OpenClaw will no longer copy `~/.codex` OAuth material into agent auth stores, so Codex users must re-authenticate via browser login or device pairing. The same release retires the `OPENCLAW_CODEX_APP_SERVER_GUARDIAN` environment variable — replace it with the plugin config `appServer.mode: "guardian"` or with `OPENCLAW_CODEX_APP_SERVER_MODE=guardian`. Codex's `serviceTier` field also stops accepting legacy values like `"priority"`; only `"fast"` and `"flex"` survive. And **v2026.4.21 makes owner-command authorization strictly identity-based**: wildcard `allowFrom` entries and empty owner-candidate lists are no longer treated as sufficient when `enforceOwnerForCommands=true`. If you ever relied on a wildcard fallback for `/exec` or similar owner-only commands, those calls will start returning permission denials immediately after upgrade — populate `commands.ownerAllowFrom` explicitly first.

## Security-driven default changes that look like bugs

Several v2026.4.20 changes will surface as mysterious authorization or env-loading failures rather than explicit errors. **All `OPENCLAW_*` environment keys are now blocked from untrusted workspace `.env` files** (fail-closed, PR #473) — operators who set runtime variables there must move them to trusted system env or to the main config. Gateway WebSocket broadcasts are gated: chat, agent, and tool-result event frames now require `operator.read` or higher, so pairing-scoped sessions and node-role sessions will stop receiving session chat passively. The agent-facing `gateway` tool guard expanded to block model-driven `config.patch` and `config.apply` against operator-trusted paths (sandbox, plugin trust, gateway auth/TLS, hook routing/tokens, SSRF policy, MCP servers, workspace filesystem hardening, including per-agent overrides under `agents.list[]`). MCP stdio servers can no longer set `NODE_OPTIONS` or similar interpreter env keys at startup. Codex transport normalization automatically rewrites stale `openai-completions` overrides on default OpenAI/Codex hosts back to the native Codex Responses transport, and the Codex Responses endpoint moved from the removed `/backend-api/responses` alias to `/backend-api/codex`. Custom-proxy users are unaffected, but anyone who hand-edited transport overrides in 2025-era configs will see them silently rewritten.

The 4.24 release adds another tier of identity tightening that also requires action: **`openclaw matrix verify self` is a new required step** for Matrix self-device verification, because full cross-signing identity trust is now enforced. Browser-control standalone loopback auth now scopes to the resolved active gateway credential and fails closed when password mode lacks a resolved password, and `browser.request` requires `operator.admin`. The 127.0.0.1 MCP surface applies owner-only tool policy, so non-owner bearer callers can no longer see or invoke `cron`, `gateway`, or `nodes` tools.

## Configuration and CLI migrations you must run

Four legacy config patterns are now **strictly rejected at load time** in v2026.4.24 and need `openclaw doctor --fix` to migrate cleanly: **legacy flat TTS provider config blocks** (PR #56220), `secretref-env:<ENV_VAR>` marker strings on SecretRef paths (#51794), malformed persisted cron job IDs from legacy `jobId` or non-string `id` fields (#70128), and stale per-agent `openai-codex:default` OAuth profiles that shadow newer identity-scoped main-agent profiles (#70393). The `openclaw doctor` command itself was rewritten in 4.21 to install only the specific missing dependency for an enabled plugin (e.g. `@slack/web-api`) instead of pulling the entire core dependency set, so re-running it after the upgrade is cheap and recommended.

Cron operators get a structural surprise in **v2026.4.20**: cron storage was split into two files, with runtime execution state moving to `jobs-state.json` so `jobs.json` stays stable for git-tracked job definitions (#63105). Anyone version-controlling cron should update `.gitignore` patterns. The same release made **Sessions Maintenance run at load time by default**, pruning oversized session stores on first start — back up `sessions.json` before upgrading if you need historical session data. CI pipelines should also note that `openclaw qa suite` and `openclaw qa telegram` now fail by default on scenario failure; pass `--allow-failures` for the previous artifact-only behavior (#69122).

The 4.22 release rewrote the model-config write path: **`openclaw models auth login` now merges provider-owned default-model additions** rather than replacing `agents.defaults.models`, and migrations that genuinely need to rename keys (Anthropic → Claude CLI) must opt in with `replaceDefaultModels`. New `config set --merge` and `--replace` flags surface the same distinction for provider-scoped allowlists. In 4.24, `openclaw models list` became **read-only** and no longer rewrites per-agent `models.json` (#70847), and provider runtime selection collapsed to one model provider choice per provider — Codex, Claude CLI, and Gemini CLI execution moved into explicit runtime selection, with legacy runtime references now fallback-only.

## Defaults that quietly change behavior and cost

Several default changes will affect output, latency, and bills without producing errors. The table below summarizes the ones most likely to surprise operators.

| Default | Old (≤4.15) | New (≤4.24) | First appeared |
|---|---|---|---|
| Onboarding model | `gpt-4o` / Claude Opus 4.6 | **DeepSeek V4 Flash** | 4.24 |
| Bundled image-gen model | older OpenAI image | **`gpt-image-2`** | 4.21 |
| Anthropic Claude Opus 4.7 context window | 200k fallback | **1M tokens** | 4.22 |
| Reasoning thinking level (no explicit config) | legacy `off`/`low` | **provider `medium` equivalent** | 4.22 |
| Discord auto-thread parent transcript | inherited | **isolated unless `channels.discord.thread.inheritParent`** | 4.22 |
| GitHub Copilot onboarding default | `gpt-4o` | **`claude-opus-4.6`** | 4.20 |
| Moonshot/Kimi default | `kimi-k2.5` | **`kimi-k2.6`** | 4.20 |
| Codex `gpt-5.5` route | OpenAI API key | **ChatGPT OAuth (`openai-codex/gpt-5.5`)** | 4.24 |
| `browser.actionTimeoutMs` | shorter | **60s** | 4.24 |
| Codex app-server approvals | permissive | **`on-request`** | 4.20 |
| Image-gen `model=` overrides | could fall through | **exact-only (no fallback to Gemini)** | 4.24 |

The reasoning-level bump and the Opus 4.7 context expansion in 4.22 are the most likely sources of unexpected token-cost increases; if you have a strict budget posture, set explicit `agents.defaults` thinking levels and context caps before upgrading rather than after.

## Plugin and dependency surface to audit before upgrading

Beyond the headline removals, v2026.4.24 reshapes the plugin system in ways that affect operators and plugin authors alike. New manifest contracts — `channelConfigs`, `modelCatalog`, and `setup.requiresRuntime: false` — let plugins declare provider-owned model rows, aliases, suppression rules, and descriptor-only setup without loading runtime code (#71342). A central plugin compatibility registry with **dated migration metadata** now tracks legacy harness naming and plugin-facing aliases on a roughly three-month removal calendar, so any plugin still using `providerAuthEnvVars` compatibility metadata or other legacy SDK surfaces will start emitting warnings. Bundled plugin runtime deps are now staged in an external runtime root for packaged and global installs, which avoids package-tree update races but means custom wrappers that introspect `node_modules` paths need updating. The bundled Pi runtime moved from 0.66.x through **0.70.2**, carrying along a renamed lifecycle event pair — `compaction_start`/`compaction_end` (#67713) — that will silently break any custom listener still using the old event names.

A few infrastructure dependencies are worth pinning down explicitly. The `uuid` transitive override is pinned to 14.0.0 for a security advisory in 4.22, and `node-domexception` was added as a `package.json` `overrides` alias in 4.21 to silence the deprecated `google-auth-library → gaxios → node-fetch → fetch-blob` warning chain. The TUI dropped `cli-highlight` in 4.24 (themed code coloring is preserved without it). On Linux, gateway-managed child processes are now wrapped in a `/bin/sh` shim that raises child `oom_score_adj`; opt out with `OPENCLAW_CHILD_OOM_SCORE_ADJ=0` if you have a custom OOM policy (#70404, #70419).

## Items the task asked about that did *not* land in this range

A few changes commonly attributed to this window actually live just outside it and should not be expected as part of a 4.15 → 4.24 upgrade:

- **`openclaw migrate` as a standalone command** does not exist in 4.15–4.24. The supported migration path in this range is `openclaw doctor --fix` plus `openclaw update`. The dedicated `openclaw migrate` command (with plan, dry-run, JSON, pre-migration backup, and Hermes importer) ships in **v2026.4.26**.
- **The cold persisted plugin registry switch** is a **v2026.4.25** change. The 4.24 release lays the foundation (manifest-backed model rows, lazy provider deps, descriptor-only setup) but does not flip the cold-registry default; the `OPENCLAW_DISABLE_PERSISTED_PLUGIN_REGISTRY` break-glass switch is also a 4.25 artifact.
- **`OPENCLAW_NO_AUTO_UPDATE=1`** as an auto-update kill-switch was not found in 4.15–4.24 notes (it appears in 4.25+). What 4.24 *does* change is the `openclaw update` mechanic itself: npm global updates now install into a verified temporary prefix before the package tree swaps, preventing mixed old/new installs and stale packaged files from breaking verification.
- **Chrome extension relay removal** — no explicit removal landed in 4.16–4.24. The legacy Chrome extension relay was removed earlier in v2026.3.22, and 4.24's browser work is additive (CDP-native role snapshots, `openclaw browser doctor --deep`, viewport `click-coords`, per-profile `headless`).

## Recommended upgrade sequence

The cleanest path I'd run in production is to first snapshot `~/.openclaw/workspace/` and back up `sessions.json`, then do a config audit for the items in the security section above (workspace `.env` `OPENCLAW_*` keys, `commands.ownerAllowFrom`, MCP stdio `NODE_OPTIONS`, Codex `serviceTier` values, legacy `secretref-env:` markers, flat TTS provider blocks). Re-authenticate Codex via browser login or device pairing before the upgrade so you don't lose access mid-flight, and add explicit `agents.defaults` for thinking level and context caps if cost discipline matters. Run `openclaw update`, then `openclaw doctor --fix` to absorb the four 4.24 strict-rejection migrations, then `openclaw matrix verify self` if you use Matrix. Finally, audit any in-house plugins for `registerEmbeddedExtensionFactory` calls and rewrite to `registerAgentToolResultMiddleware`; this is the only change in the window that genuinely cannot be deferred.

The upgrade is comfortably tractable, but it is not the kind you do at 5pm on a Friday. The 4.20–4.24 stretch is the most aggressive identity-and-trust tightening OpenClaw has shipped in a single quarter, and the payoff — strict owner authorization, scoped gateway broadcasts, descriptor-only plugin setup, lazy provider deps — is worth the migration tax only if you actually run the doctor pass and re-validate plugin trust afterwards.
